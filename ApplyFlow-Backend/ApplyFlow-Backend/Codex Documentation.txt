Below is documentation for the **current implementation only**. No new code.

**Beginner View**
This backend is a small ASP.NET Core Web API. It exposes authentication endpoints, stores users/applications in PostgreSQL through Entity Framework Core, hashes passwords with BCrypt, and returns JWT tokens after registration/login.

Current main flow:

1. Client calls an API endpoint.
2. ASP.NET Core routes the request to a controller.
3. Controller calls `AuthService` for auth logic.
4. `AuthService` reads/writes users using `AppDbContext`.
5. EF Core talks to PostgreSQL.
6. Response DTOs are returned to the client.
7. Protected endpoints require a valid JWT in the `Authorization: Bearer <token>` header.

**Request Lifecycle**
Beginner version: request comes in, controller receives it, service does the real work, database saves/reads data, response goes back.

Technical version:

1. Kestrel receives HTTP request.
2. Middleware pipeline runs: HTTPS redirection, authentication, authorization.
3. Endpoint routing maps request to controller action.
4. Model binding creates DTO object from JSON body.
5. `[ApiController]` performs automatic validation for DataAnnotations.
6. Controller calls `IAuthService`.
7. `AuthService` uses scoped `AppDbContext`.
8. EF Core translates LINQ to SQL for PostgreSQL.
9. Service returns DTO.
10. Controller returns HTTP result.

**Authentication Flow**
Register:

1. `POST /api/auth/register`
2. Body maps to `RegisterRequestDto`.
3. `AuthController.Register()` calls `AuthService.RegisterAsync()`.
4. Email is trimmed/lowercased.
5. Service checks if email already exists.
6. Password is hashed with BCrypt.
7. User is saved.
8. JWT is generated.
9. Response returns token + user data.

Login:

1. `POST /api/auth/login`
2. Body maps to `LoginRequestDto`.
3. Service finds user by normalized email.
4. BCrypt verifies password against stored hash.
5. If valid, JWT is generated.
6. If invalid, `401 Unauthorized`.

**JWT Flow**
Beginner version: JWT is a signed login pass. Client stores it and sends it with protected requests.

Technical version:

- Generated in `AuthService.GenerateJwtToken()`.
- Claims included:
  - `sub`: user id
  - `email`: user email
  - `ClaimTypes.NameIdentifier`: user id
  - `ClaimTypes.Email`: user email
- Signed with HMAC SHA256 using `Jwt:Key`.
- Validated by JWT Bearer middleware using:
  - issuer validation
  - audience validation
  - lifetime validation
  - signing key validation
- Protected endpoints use `[Authorize]`.

**Password Hashing Flow**
- Plain password is never saved.
- `BCrypt.Net.BCrypt.HashPassword(request.Password)` creates a salted hash.
- Stored value goes into `Users.PasswordHash`.
- Login uses `BCrypt.Net.BCrypt.Verify(inputPassword, storedHash)`.
- BCrypt automatically handles salt and work factor internally.

**Entity Framework Flow**
- `AppDbContext` is registered with `AddDbContext`.
- Lifetime is scoped: one context per request scope.
- `DbSet<User>` maps to `Users`.
- `DbSet<Application>` maps to `Applications`.
- LINQ queries like `AnyAsync`, `FirstOrDefaultAsync`, `SingleAsync` become SQL.
- `SaveChangesAsync()` commits inserts/updates.

**Database Schema**
`Users` table:

| Column | Type | Notes |
|---|---|---|
| `Id` | `uuid` | Primary key |
| `Email` | `varchar(255)` | Required, unique |
| `PasswordHash` | `varchar(255)` | Required, BCrypt hash |
| `CreatedAt` | timestamp with time zone | Required |

`Applications` table:

| Column | Type | Notes |
|---|---|---|
| `Id` | `uuid` | Primary key |
| `UserId` | `uuid` | Indexed, but no foreign key currently |
| `CompanyName` | `varchar(255)` | Required |
| `PositionTitle` | `varchar(255)` | Required |
| `Status` | `varchar(50)` | Required |
| `AppliedDate` | timestamp with time zone | Required |

Important: `Applications.UserId` has an index but no configured relationship/foreign key to `Users`.

**Security Mechanisms**
Implemented:

- BCrypt password hashing.
- JWT Bearer authentication.
- JWT issuer/audience/lifetime/signature validation.
- `[Authorize]` on `/api/users/me`.
- DTOs prevent returning password hashes.
- Unique email database index.
- Minimum JWT key length check.
- Register DTO requires valid email and password length >= 8.

Not yet implemented:

- Refresh tokens.
- Role-based authorization.
- Rate limiting.
- Account lockout.
- Email verification.
- HTTPS enforcement exists via middleware, but production HTTPS/proxy setup still needs deployment config.
- JWT secret is stored in `appsettings.json`; production should use environment variables or secret manager.

Scoped means a new instance is created per request scope.

**Middleware Pipeline**
Current order:

1. Development only: `UseSwagger()`
2. Development only: `UseSwaggerUI()`
3. `UseHttpsRedirection()`
4. `UseAuthentication()`
5. `UseAuthorization()`
6. `MapControllers()`
7. `Run()`

Authentication must come before authorization, and it does.

**Program.cs Execution Flow**
1. Create builder.
2. Register MVC controllers.
3. Register Swagger/OpenAPI and Bearer security definition.
4. Register PostgreSQL `AppDbContext`.
5. Register `IAuthService`.
6. Read and validate JWT key.
7. Configure JWT Bearer token validation.
8. Build app.
9. Enable Swagger in development.
10. Enable HTTPS redirection.
11. Enable auth middleware.
12. Map controllers.
13. Start web server.

**AppDbContext Responsibilities**
Beginner: it is the bridge between C# objects and PostgreSQL tables.

Technical:

- Exposes `Users` and `Applications` DbSets.
- Configures primary keys.
- Configures required fields.
- Configures max lengths.
- Configures unique email index.
- Configures `Applications.UserId` index.
- Does not currently configure navigation properties or FK relationships.

**Controller Responsibilities**
`AuthController`:
- Receives register/login requests.
- Delegates logic to `IAuthService`.
- Converts service exceptions into HTTP responses.

`UsersController`:
- Requires JWT for `/me`.
- Reads authenticated user id from claims.
- Calls `IAuthService.GetCurrentUserAsync()`.

`WeatherForecastController`:
- Template demo endpoint only.

`ApplicationsController`:
- Currently not a real API controller for job applications.
- It inherits `Controller` and returns `View()`, which is more MVC than Web API.

**Service Responsibilities**
`AuthService` owns the auth business rules:

- Normalize email.
- Check duplicate email.
- Hash password.
- Verify password.
- Save users.
- Generate JWT.
- Map `User` entity to safe response DTO.
- Fetch current user without tracking.

**DTO Responsibilities**
DTOs protect the API boundary.

- `RegisterRequestDto`: input for account creation.
- `LoginRequestDto`: input for login.
- `AuthResponseDto`: token + expiry + safe user data.
- `UserResponseDto`: safe public user shape with no password hash.

**How Data Travels**
Register example:

```text
JSON body
 -> RegisterRequestDto
 -> AuthController.Register
 -> AuthService.RegisterAsync
 -> AppDbContext.Users.AnyAsync
 -> BCrypt hash
 -> AppDbContext.Users.Add
 -> SaveChangesAsync
 -> AuthResponseDto
 -> HTTP 201 response
```

Login example:

```text
JSON body
 -> LoginRequestDto
 -> AuthController.Login
 -> AuthService.LoginAsync
 -> AppDbContext.Users.FirstOrDefaultAsync
 -> BCrypt.Verify
 -> JWT generation
 -> AuthResponseDto
 -> HTTP 200 response
```

Current user example:

```text
Authorization header
 -> JWT middleware validates token
 -> ClaimsPrincipal is created
 -> UsersController.Me
 -> ClaimTypes.NameIdentifier read
 -> AuthService.GetCurrentUserAsync
 -> AppDbContext.Users.AsNoTracking
 -> UserResponseDto
 -> HTTP 200 response
```

**Current Limitations And Technical Debt**
- `ApplicationsController` is still a placeholder, not production API CRUD.
- `Application` model has no EF relationship to `User`.
- No foreign key from `Applications.UserId` to `Users.Id`.
- No refresh token flow.
- JWT secret is in `appsettings.json`, not secret storage.
- No structured error handling middleware.
- No logging inside controllers/services.
- No request/response DTOs for applications yet.
- No pagination/filtering/search for applications.
- No integration tests for API endpoints.
- No production CORS policy.
- No email verification.
- No password complexity beyond min length.
- No account lockout/rate limiting.
- No soft delete fields.
- WeatherForecast template files still exist and should eventually be removed.
- Test project is nested inside API folder, so main `.csproj` excludes it manually. It works, but a sibling test project folder would be cleaner long-term.
